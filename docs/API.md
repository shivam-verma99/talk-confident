## 1. Conventions

| Topic | Convention |
|---|---|
| Base URL (local) | `http://localhost:8000` |
| Base URL (Android emulator → host) | `http://10.0.2.2:8000` |
| Encoding | UTF-8 |
| Content-Type | `application/json` unless noted (multipart for audio) |
| Authorization header | `Authorization: Bearer <jwt>` |
| Times | ISO 8601 with timezone (e.g. `2026-05-26T11:30:00+00:00`) |
| IDs | UUID v4 strings (e.g. `e72b…-…-…-…`) |
| Scores | Integers 0–100 |
| Booleans | JSON `true` / `false` (never `1`/`0`) |
| Errors | `{ "detail": "<message>" }` — see [§7](#7-error-model) |

All endpoints except `POST /auth/google` and `GET /healthz` require the bearer
token in the `Authorization` header.

## 2. Authentication

The backend uses **Google Sign-In only** — there is no password, no email
registration, no SMTP. The flow:

1. The Android client obtains a Google **ID token** (NOT an access token) via the
   Credential Manager API / Google Identity Services SDK. The OAuth client must
   be an **Android** client in the same Google Cloud project that the backend's
   `GOOGLE_OAUTH_CLIENT_ID` points to.
2. The client posts that ID token to `POST /auth/google`.
3. The backend verifies the ID token against Google's public keys (signature,
   audience, expiry, `email_verified`).
4. On first sight the backend creates a `User` + `Profile` row keyed by Google's
   `sub`. On subsequent sign-ins it updates `last_login_at`.
5. The backend returns a JWT (`access_token`) the client must store and attach
   to every subsequent call.

**Token lifetime.** JWTs are signed with HS256 and valid for `JWT_EXPIRES_MINUTES`
(default **30 days**, returned in `expires_in` as seconds). When the token
expires, the client must redo the Google ID-token exchange.

**Logout.** No server-side state — discard the JWT on the client.

**Errors:**

| Status | When |
|---|---|

### 3.3 `GET /curriculum/next`

Return today's tailored vocabulary + practice sentences, generated by Gemini

**Auth:** Bearer JWT.

**Query:**

| Param | Type | Default | Description |
|---|---|---|---|
| `mode` | `read_aloud` \| `meeting_prep` | `read_aloud` | Hint to the curriculum generator. `meeting_prep` raises the chance of a `meeting_scenario_seed`. |

**Response 200:**

```json
{
  "level": 3,
  "focus_areas": ["fluency under pressure", "th-sound clarity"],
  "words": [
    {
      "word": "escalate",
      "definition": "raise an issue to higher authority",

      "why_chosen": "vendor escalation phrasing weakness"
```

### 3.4 `POST /practice/evaluate`

Evaluate a single read-aloud or spontaneous practice attempt. The audio file is
sent in a multipart upload; the server streams it through to Gemini and **never
writes it to disk**.

**Auth:** Bearer JWT.
**Content-Type:** `multipart/form-data`.

**Form fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `audio` | file | yes | Recorded audio in one of the supported MIME types (see [§5](#5-audio-handling)). |
| `target_text` | string | no | The text the user was asked to read aloud. Omit for spontaneous practice. |
| `mode` | `read_aloud` \| `spontaneous` | yes | Must NOT be `meeting_prep` — use the meeting endpoints for that. |
| `client_audio_ref` | string | recommended | Opaque ID identifying the audio file on the device (e.g. the local file UUID). Stored alongside the attempt so the device can correlate scores with its local audio library. |
| `duration_seconds` | float | no | Length of the recording. Stored for display only — server does not validate. |
| `session_id` | UUID | no | Link this attempt to a pre-existing session. |

**Response 201:**

```json
{
  "attempt_id": "9c0e1d...",
  "session_id": null,
  "composite_score": 71,
  "quality_band": "neutral",
  "evaluation": {
    "transcript": "We need to escalate this to the vendor.",
    "scores": {
      "pronunciation_clarity": 72,
      "fluency": 68,
      "confidence": 65,
      "grammar": 80
    },
    "hesitation_markers": [
      { "type": "filler", "timestamp_s": 1.4, "note": "uh before 'vendor'" }
    ],
    "weaknesses": ["th-sound substitution in 'this'"],
    "strengths": ["clear sentence-final intonation"],
    "corrections": [
      {
        "issue": "'this' pronounced as 'dis'",
        "fix": "Bite tongue lightly for /th/ sound.",
        "example": "Place 'this' on your tongue."
      }
    ],
    "recommended_drill": {
      "name": "th-sound minimal pairs",
      "instructions": "Practise 'this/dis', 'three/tree' aloud 10 times."
    },
    "overall_note": "Steady pace, control filler words."
  }
}
```

**Errors:**

| Status | Reason |
|---|---|
| `400` | Bad audio (unsupported MIME, empty, oversized) or `mode=meeting_prep` (use the meeting endpoints). |
| `401` | Missing/invalid token. |
| `429` | Gemini rate-limited after retries. `Retry-After` header is set. |
| `502` | Gemini failed to produce a valid evaluation. |

**Side effects:**

- A `PracticeAttempt` row is created with full scores, transcript, and the
  `client_audio_ref`.
- The user's `readiness_score` is recomputed.
- If the rolling token cost crosses `CONTEXT_TOKEN_THRESHOLD`, a background
  summarization job is queued — the client sees no change in this response.

### 3.5 `POST /practice/meeting-prep/start`

Open a Meeting Prep session. The backend generates a BSNL-flavored scenario
(fiber-cut restoration, vendor escalation, OFC route survey, etc.) tailored to
the user's recent weaknesses.

**Auth:** Bearer JWT.

**Request:**

```json
{
  "focus_area": "fiber_cut_restoration"
}
```

`focus_area` is optional. If omitted, the model picks a topic that addresses the
user's current weaknesses.

**Response 201:**

```json
{
  "session_id": "1b56...",
  "mode": "meeting_prep",
  "scenario": {
    "title": "Fiber-cut restoration vendor escalation",
    "context": "Late-evening fiber cut between Bikaner exchange and Jaipur junction. Vendor has been on-site 45 minutes with no ETA. SLA breach in 30 minutes.",
    "your_role": "Divisional Engineer (Maintenance)",
    "counterpart_role": "Vendor Field Engineer",
    "objectives": [
      "Get a concrete restoration ETA.",
      "Confirm spare splicing inventory.",
      "Document the cause for the morning briefing."
    ],
    "opening_prompt": "Sir, we are still pulling the new patch. Can you confirm the route diversion?",
    "suggested_phrases": [
      "Please give me a concrete ETA.",
      "What is your current blocker?"
    ]
  }
}
```

The client should render `context`, `your_role`, `counterpart_role`,
`objectives`, and display `opening_prompt` as the **counterpart's first line**.
The user then records their reply and posts it to `/turn` (next endpoint).

**Errors:** `401`, `502`.

### 3.6 `POST /practice/meeting-prep/turn`

Submit one spoken turn inside an active Meeting Prep session. The backend
evaluates the user's speech AND generates the counterpart's next reply so the
role-play continues.

**Auth:** Bearer JWT.
**Content-Type:** `multipart/form-data`.

**Form fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `audio` | file | yes | Recording of this turn. |
| `session_id` | UUID | yes | The session returned by `/start`. |
| `client_audio_ref` | string | recommended | Same convention as `/evaluate`. |
| `duration_seconds` | float | no | Length of the recording. |

**Response 201:**

```json
{
  "attempt_id": "4a92...",
  "session_id": "1b56...",
  "composite_score": 68,
  "quality_band": "neutral",
  "evaluation": {
    "transcript": "Yes, route diversion is approved. Please share the ETA.",
    "scores": { "pronunciation_clarity": 70, "fluency": 66, "confidence": 62, "grammar": 78 },
    "hesitation_markers": [],
    "weaknesses": ["soft sentence endings"],
    "strengths": ["calm tone"],
    "corrections": [
      {
        "issue": "Trailing voice on 'ETA'",
        "fix": "Land 'ETA' with downward intonation.",
        "example": "Share the ETA."
      }
    ],
    "recommended_drill": {
      "name": "End-of-sentence pitch drop",
      "instructions": "Read 5 sentences ending with a noun, dropping pitch on the last syllable."
    },
    "overall_note": "Clear ask, but voice fades at the end."
  },
  "next_prompt": "ETA is 20 minutes once the splicing kit arrives."
}
```

`next_prompt` is the counterpart's next line — display it as **dialogue**, not
feedback. The evaluation block is the coaching feedback for the just-spoken
turn.

**Errors:**

| Status | Reason |
|---|---|
| `400` | Bad audio, or session already closed. |
| `401` | Missing/invalid token. |
| `404` | `session_id` not found or not owned by this user. |
| `502` | Gemini failure. |

### 3.8 `GET /practice/attempts`

List recent practice attempts so the Android app can curate its local audio
library and surface the best / worst clips.

**Auth:** Bearer JWT.

**Query:**

| Param | Type | Default | Description |
|---|---|---|---|
| `band` | `all` \| `best` \| `worst` \| `neutral` | `all` | Filter by quality band. |
| `limit` | int (1–200) | 50 | Page size. |
| `offset` | int (≥0) | 0 | Page offset. |

**Response 200:** Array of `AttemptOut` objects, newest first.

```json
[
  {
    "id": "9c0e1d...",
    "session_id": null,
    "target_text": "We need to escalate this to the vendor.",
    "transcript": "We need to escalate this to the vendor.",
    "composite_score": 78,
    "pronunciation_score": 80,
    "fluency_score": 76,
    "confidence_score": 72,
    "grammar_score": 88,
    "quality_band": "best",
    "client_audio_ref": "local-001",
    "duration_seconds": 4.20,
    "mime_type": "audio/wav",
    "created_at": "2026-05-26T11:32:14+00:00"
  }
]
```

**Typical client use:**

```text
1. After every /practice/evaluate, store the response (esp. quality_band)
   keyed by client_audio_ref in your local DB.
2. On a "best clips" screen, call GET /practice/attempts?band=best and join
   the returned client_audio_ref values with your local audio files.
3. Same for worst, to help the user re-listen and re-practise weak takes.
4. Periodically delete neutral clips your user has reviewed; keep best/worst
   if they want to retain them.
```

## 5. Audio handling

### Server contract

- **The server never stores raw audio.** Uploads stream into a bounded in-memory
  buffer, go directly to Gemini, and are dropped immediately after the response.
- If the upload is **≥ 20 MB**, the server transparently routes it through
  Gemini's Files API and deletes the file handle in a `finally` block.

### Client contract

| Concern | Recommendation |
|---|---|
| **Format** | WAV PCM 16-bit mono 16 kHz is small, accurate, and well-supported by Gemini. AAC/M4A/MP3/OGG are also accepted. |
| **Allowed MIME types** | `audio/wav`, `audio/x-wav`, `audio/mpeg`, `audio/mp3`, `audio/mp4`, `audio/aac`, `audio/ogg`, `audio/webm`, `audio/flac`. |
| **Hard upload limit** | 50 MB (server returns 400 above). |
| **Recommended length** | 5–60 seconds for read-aloud; up to ~3 minutes for meeting turns. Anything > 2 minutes will hit the Files API path automatically. |
| **`client_audio_ref`** | Generate a stable, unique string per local file (e.g. the file's UUID). Send it with every evaluation. The server stores ONLY this string; the audio bytes live in your app-private storage. |
| **Persistence on device** | Your responsibility. Suggested layout: `audio/<client_audio_ref>.wav` in app-private storage; a Room table that joins `client_audio_ref` to the latest `EvaluationResponse`. |
| **Deletion** | Always delete the local audio file when the user removes an attempt. The server only holds metadata. |

### Multipart upload example (Kotlin / OkHttp / Retrofit)

```kotlin
interface PracticeApi {
    @Multipart
    @POST("practice/evaluate")
    suspend fun evaluate(
        @Header("Authorization") auth: String,
        @Part audio: MultipartBody.Part,
        @Part("target_text") targetText: RequestBody?,
        @Part("mode") mode: RequestBody,
        @Part("client_audio_ref") clientAudioRef: RequestBody,
        @Part("duration_seconds") durationSeconds: RequestBody?,
    ): EvaluationResponse
}

val audioPart = MultipartBody.Part.createFormData(
    "audio", file.name, file.asRequestBody("audio/wav".toMediaType())
)
```

## 6. Best / Worst surfacing

The server classifies every attempt into a `quality_band`:

- `best` — composite score is at or above the rolling p75 of the last 30
  attempts AND ≥ median + 8.
- `worst` — at or below p25 AND ≤ median − 8.
- `neutral` — anything in between, including the user's first 4 attempts
  (insufficient signal).

Use this directly in the UI:

1. **List screen.** Tag clips visually based on `quality_band`. Best = green,
   worst = amber. Neutral = no tag.
2. **Filtered views.** Add tabs/chips that call
   `GET /practice/attempts?band=best` and `?band=worst`.
3. **Auto-curation.** When the local audio library gets large, prompt the user
   to delete `neutral` clips while protecting `best` and `worst`.

`composite_score` is also returned on every attempt — use it for sort order
within a band if you want a "top 10 best ever" feature.

## 7. Error model

Every error response is a single object:

```json
{ "detail": "Audio payload exceeds limit of 52428800 bytes." }
```

Status codes used by this API:

| Status | Meaning |
|---|---|
| `200` | OK. |
| `201` | Resource created (evaluations, sessions). |
| `204` | No content (close session). |
| `400` | Validation failed — bad MIME, empty audio, wrong mode for endpoint, etc. |
| `401` | Auth failure — missing token, invalid JWT, Google ID-token rejected. The `WWW-Authenticate: Bearer` header is set on most variants. |
| `404` | Resource not found or not owned by the current user. |
| `422` | Pydantic / request-shape validation failure (FastAPI default). The body contains a `detail` array with field-level errors. |
| `429` | Gemini rate-limited after retries. Includes `Retry-After` header (seconds). Back off and retry. |
| `502` | Gemini failed to produce a valid response (e.g. malformed JSON after retries). Safe to retry; if it persists, report to the user. |
| `503` | Server didn't initialise the Gemini client. Configuration issue — page the team. |

### Pydantic 422 example

```json
{
  "detail": [
    {
      "type": "missing",
      "loc": ["body", "id_token"],
      "msg": "Field required",
      "input": {}
    }
  ]
}
```

## 8. Rate limits, retries, and timeouts

- **Server side.** Every Gemini call is wrapped in tenacity with up to **3
  attempts** and exponential back-off (2s → 10s). The client doesn't need to
  retry transient Gemini errors itself.
- **429 from server.** Honor `Retry-After` (seconds). Don't tight-loop.
- **Timeouts.** A typical evaluation takes 1.5–4 s. Set the client's HTTP
  timeout to **30 s** for `/practice/evaluate` and meeting endpoints, 60 s for
  the first call after a long idle period (the persona cache may need to be
  rebuilt). 10 s is plenty for everything else.
- **Concurrency.** Don't send a second evaluation while the previous one is
  in-flight for the same user — sessions and readiness updates are easier to
  reason about serialised.

## 9. Android integration checklist

- [ ] Google Cloud project has an **Android OAuth client** with your package
      name + SHA-1; backend `GOOGLE_OAUTH_CLIENT_ID` matches.
- [ ] App uses Credential Manager (`GetCredentialRequest` with
      `GetGoogleIdOption`) to obtain an ID token.
- [ ] POST that token to `/auth/google`; persist the resulting JWT securely
      (EncryptedSharedPreferences or `androidx.security`).
- [ ] Attach `Authorization: Bearer <jwt>` to every subsequent call.
- [ ] Record audio in 16 kHz mono WAV by default; switch to AAC for longer
      monologues if file size becomes a concern.
- [ ] Generate a per-recording `client_audio_ref` (UUID) and store it alongside
      the local file path. Send it with every evaluation.
- [ ] After receiving `EvaluationResponse`, persist scores and `quality_band`
      keyed by `client_audio_ref`. Surface band in the audio list UI.
- [ ] On a "best" or "worst" filter, hit `GET /practice/attempts?band=…` and
      join with local audio files.
- [ ] Refresh `GET /user/progress` after every N evaluations (e.g. 5) or when
      the user opens the progress tab; render the level-up nudge respectfully
      (never auto-advance).
- [ ] Treat `recommendation.should_level_up=true` as a suggestion, not a
      command. The user picks when to actually advance.
- [ ] Handle `401` globally by clearing the JWT and triggering a re-login.

## 10. OpenAPI / Swagger

The backend ships an auto-generated, always-current OpenAPI document:

| Where | URL |
|---|---|
| OpenAPI JSON | `GET /openapi.json` |
| Swagger UI | `GET /docs` |
| ReDoc | `GET /redoc` |

Point your OpenAPI generator (e.g. `openapi-generator-cli`) at
`/openapi.json` to generate a typed Kotlin client. Recommended template:
`kotlin` with `library=jvm-retrofit2,coroutines`. Use the schemas in this
document as the human-readable companion to the auto-generated client.
